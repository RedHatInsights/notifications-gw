# Logging and Observability Guidelines

## Logging Framework

- Use `io.quarkus.logging.Log` (static import) for all application logging. This codebase does not use SLF4J or `java.util.logging` directly in application code, except that `NotificationsGwApp` uses `java.util.logging.Logger` internally to install the access log filter — do not use it elsewhere.
- Use the `f`-suffixed format methods (`Log.infof`, `Log.errorf`, `Log.warnf`, `Log.debugf`, `Log.tracef`) with `%s`/`%d` placeholders, not string concatenation.
- When logging exceptions, pass the exception as the first argument to the format method (e.g., `Log.errorf(e, "message: %s", detail)`), not as a trailing argument.

## Log Levels

- Use `Log.error` for failures that affect message delivery or backend communication (Kafka delivery failures, unreachable `notifications-backend`).
- Use `Log.warn` for rejected requests that indicate policy violations (e.g., staging allow-list rejections in `GwResource`).
- Use `Log.info` for startup configuration values, cache refresh results, and authentication validation outcomes. Follow the pattern in `GwConfig.logConfiguration()` for new config properties.
- Use `Log.debug` for validation results (event type lookups, bad-request responses from backend).
- Use `Log.trace` for request-level details. Guard trace-level logging with `Log.isTraceEnabled()` when the log message requires computation, as done in `IncomingRequestInterceptor`.

## Log Injection Prevention

- Sanitize user-controlled strings before logging them at trace level. Use the `ANTI_INJECTION_PATTERN` approach from `IncomingRequestInterceptor` (strip `\n`, `\r`, `\t`) when logging URI paths or other external input.
- Prefer logging structured fields (`bundle=%s, application=%s, orgId=%s`) rather than raw request bodies. Follow the bracket-delimited format used in `GwResource` warn/debug messages.

## Log Configuration

- The application log category is `com.redhat.cloud.notifications` (set to INFO in prod, DEBUG in test via `%test.` prefix in `application.properties`).
- Access logs use the `access_log` category with the `combined` pattern. The `NotificationsGwApp.initAccessLogFilter()` method filters out successful `/health` and `/metrics` requests to reduce noise.
- Prefer adjusting log levels via the `QUARKUS_LOG_CATEGORY__COM_REDHAT_CLOUD_NOTIFICATIONS__LEVEL` env var in `clowdapp.yaml` rather than changing `application.properties`.

## External Log Sinks

- CloudWatch logging (`quarkus-logging-cloudwatch`) is disabled by default. It streams to the `notificationsLogGroup` log group with `notifications-gw` as the stream name. In production, the stream name is overridden to `${HOSTNAME}` via the ClowdApp template.
- Sentry logging (`quarkus-logging-sentry`) is disabled by default. It filters to `com.redhat.cloud.notifications` packages via `quarkus.log.sentry.in-app-packages`. The DSN includes the environment name suffix.
- Avoid enabling these sinks in `application.properties` directly; they are toggled via `QUARKUS_LOG_CLOUDWATCH_ENABLED` and `QUARKUS_LOG_SENTRY_ENABLED` env vars in deployment.

## Metrics (Micrometer/Prometheus)

- This project uses `quarkus-micrometer-registry-prometheus`. Metrics are exposed at `/metrics` (not `/q/metrics` — `quarkus.http.non-application-root-path=/` is set in `application.properties`).
- Three custom counters exist in `GwResource`:
  - `notifications.gw.received` -- incremented on every incoming POST to `/notifications`.
  - `notifications.gw.forwarded` -- incremented on successful Kafka delivery.
  - `notifications.gw.failed.requests` -- incremented on failures, tagged with `status_code` (e.g., `400`, `503`).
- Register counters at construction time via `MeterRegistry` constructor injection, as `GwResource` does. For dynamic tags (like `status_code`), use `meterRegistry.counter(name, Tags.of(...))` at increment time.
- Follow the `notifications.gw.` prefix for new metric names.

## Health Checks

- `quarkus-smallrye-health` exposes `/health/live` and `/health/ready`. The ClowdApp template configures Kubernetes probes against port 8000 with 40-second initial delay.
- The access log filter in `NotificationsGwApp` suppresses successful health and metrics endpoint logs using `FILTER_REGEX`. If you add new internal endpoints that should be filtered from access logs, extend that regex and add test cases in `NotificationsGwAppTest`.

## Startup Logging

- Log configuration values at startup via a `logConfiguration()` method on config beans, called from `NotificationsGwApp.init()`. Follow the `GwConfig.logConfiguration()` pattern: one `Log.infof` per config property using the format `"notifications.some-property = %s"` with the value as the argument.
- The git commit abbreviation is logged at startup from `git.properties` (generated by `git-commit-id-maven-plugin`).

## Fault Tolerance Observability

- All REST client methods in `RestValidationClient` and `RestInternalClient` use `@Retry(maxRetries = 5)`. Micrometer automatically tracks retry metrics via the `quarkus-smallrye-fault-tolerance` integration. Avoid adding custom retry-count metrics that duplicate this.

## Verification

```bash
# Confirm only io.quarkus.logging.Log is used (no SLF4J, no java.util.logging in app code)
grep -rn "import org.slf4j\|import java.util.logging.Logger" src/main/java/com/redhat/cloud/notifications/ --include="*.java" | grep -v NotificationsGwApp.java

# Verify metric name prefix consistency
grep -rn "counter\|gauge\|timer" src/main/java/ --include="*.java" | grep -v "import\|test"

# Run tests including access log filter tests
./mvnw test --no-transfer-progress

# Check for log injection risks -- trace/debug logging of external input without sanitization
grep -rn "Log.tracef\|Log.debugf" src/main/java/ --include="*.java"
```
